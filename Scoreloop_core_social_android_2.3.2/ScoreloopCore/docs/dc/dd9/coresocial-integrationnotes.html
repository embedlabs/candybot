Notes on Integration

The Scoreloop Programming Model

All relevant entities are exposed as model classes, including:

  • User,
  • Score,
  • Challenge,
  • Award,
  • Achievement,
  • Activity.

See the API documentation for more detailed descriptions of individual model classes.

Model classes are generally managed by two data controllers:

  • a controller that manages single instances of the class,
  • a controller that manages lists of model objects.

Scoreloop differentiates between both controllers using the singular and plural forms of the model class. Therefore:

  • the UserController manages single instances of the User class,
  • the UsersController manages a list of User objects.

Controllers will also generally have an associated observer that will receive notifications and callbacks from the server in certain circumstances. All Scoreloop observers derive from the RequestControllerObserver base class.

Please look at the API documentation for mode detailed information regarding the model, controller, and observer classes.

Some Recommendations on Integration

  • CoreSocial APIs should be called exclusively on the Main Thread (a.k.a. UI-Thread). The API is designed to be asynchronous and uses an internal worker thread for its communications with the Scoreloop server. So, the Main Thread will not be blocked even though the API call is in progress. When a request is ready, Scoreloop ensures, that the callback methods are called again on the Main Thread, so that you can immediately update the UI or state additional CoreSocial calls. Even if your game uses additional threads, please do not call CoreSocial APIs directly from them. In these cases you would have to use Android Handler objects (or equivalent) to call the CoreSocial API from the Main Thread - as you would do when performing common UI tasks.
  • Initialization of Scoreloop should take place in the Init method of your application. The section on Initialization provides practical examples of how to do this.
