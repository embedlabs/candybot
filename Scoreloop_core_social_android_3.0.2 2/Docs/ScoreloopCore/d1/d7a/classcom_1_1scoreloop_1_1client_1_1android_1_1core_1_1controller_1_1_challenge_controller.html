Detailed Description

The ChallengeController manages single instances of the Challenge class. The ChallengeController is used to

  • create a challenge
  • accept a challenge
  • reject a challenge
  • submit a challenge to the server

Challenge Lifecycle:

  1. The contender decides on a stake and creates a challenge. It is at this stage that they decide whether the challenge should be open or direct.
  2. The contender plays the game and submits the challenge and the score they obtain to the Scoreloop server
  3. A second Scoreloop user, the contestant, accepts the challenge and pays the challenge stake. If the challenge is open it may be accepted by any Scoreloop user. If it is a direct challenge, only the user to whom the challenge was assigned by the contender may accept it.
  4. The contestant plays the game and submits their score to Scoreloop
  5. Scoreloop compares the scores of the contender and the contestant in accordance with the game scoring schema as defined by the game developer and declares a winner
  6. The winner takes the challenge prize, (75% of the combined contender and contestant stakes)

Basic Usage:

To create a challenge:

  1. Call the createChallenge() method to create the challenge with the given stake and contestant.
  2. The challenge contender is automatically set equal to the current session user. The contestant is set equal to the user argument passed in step 1.Pass a null value as the user argument instead, in order to create an open challenge. The controller will automatically set the challenge object in the current session.

To accept, reject, or submit a challenge:

  1. Get an instance of ChallengeController, or re-use an existing instance.
  2. Set the challenge that the controller should deal with setChallenge(final Challenge). If no challenge is explicitly set, the controller will take the challenge from the current session instead.
  3. Call the appropriate method to accept, reject, or submit the challenge.
  4. Await notification of success via delegate callbacks.

Important usage note:

Do not modify a challenge object or one of its sub-objects like score or the userContext of a score while one of the following asynchronous methods is in progress as this might result in undefined results:

  • rejectChallenge()
  • acceptChallenge()
  • submitChallenge()
  • submitChallengeScore()

Member Function Documentation

void acceptChallenge ( )

Accepts the challenge on behalf of the session user. Only open challenges or those that have been directly assigned to the session user can be accepted.

When this method is called, the RequestControllerObserver will be called asynchronously.

See also:

void createChallenge ( final Money  aSomeMoney,
final User  aContestant 
)

Creates a new challenge. This method sets the session user as the challenge contender. The challenge contestant is the User object passed as a method argument. To create an open (non-assigned) challenge, pass a null value as the contestant argument. Challenges must be instantiated with a challenge stake, the size of which is chosen by the contender.

Parameters:
aSomeMoneyMoney object corresponding to the challenge stake. This value cannot be null. Valid challenge stakes can be gotten from the current session (use the current session to call Session.getChallengeStakes()).
aContestantUser object, the challenge contestant. Passing a null value here creates an open challenge that can be accepted by any player.

void submitChallenge ( )

Submits the challenge to the server. If the challenge is unknown to the server, it will be registered and assigned a unique identifier. If the challenge is already known to the server, calling this method will update the challenge details.

When this method is called, the RequestControllerObserver will be called asynchronously.
