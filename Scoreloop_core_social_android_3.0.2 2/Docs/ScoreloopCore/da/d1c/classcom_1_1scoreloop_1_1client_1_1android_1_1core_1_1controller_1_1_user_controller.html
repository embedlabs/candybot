Detailed Description

The UserController is used to retrieve information about an arbitrary user from the server. It is also used to update the attributes and buddy-list of the current session user.

Basic Usage:

a) To retrieve information for the current session user:

  1. Get an instance of UserController.
  2. Call loadUser().
  3. Await notification of success via delegate callbacks.
  4. After a successful request, access the user object via getUser().

b) To retrieve additional information for the session user:

  1. Get an instance of UserController, or re-use an existing instance.
  2. Call loadUserDetail().
  3. Await notification of success via delegate callbacks.
  4. After a successful request, access the user object via getUser().
  5. Use the user object to call User.getDetail() to access the additional information.

c) To retrieve information for an arbitrary user:

  1. Get an instance of UserController, or re-use an existing instance.
  2. Set the user property of the controller by calling setUser() so that it equals the user for whom details are to be requested.
  3. Call loadUser().
  4. Await notification of success via delegate callbacks.
  5. After a successful request, access the user object via getUser().

d) To add a user to the buddy-list of the session user:

  1. Set the user property of the controller equal to the buddy to be added. Use setUser() to do this.
  2. Call addAsBuddy().
  3. Await notification of success via delegate callbacks.

e) To load the buddy-list of a user:

  1. Set the user property equal to the user for whom the buddy-list will be requested. Call setUser() to do this. If no user is explicitly set, the buddy-list of the session user will be requested.
  2. Call loadBuddies() to make the server request.
  3. Await notification of success via delegate callbacks.
  4. After a successful request access the user object via getUser().
  5. Use the user object to access the buddy-list, (by calling User.getBuddyUsers()).

Important usage note:

Do not modify a user object or one of its sub-objects, like context, while one of the following asynchronous methods is in progress, as this might give undefined results:

  • loadBuddies()
  • loadUser()
  • loadUserContext()
  • loadUserDetail()
  • submitUser()
  • submitUserContext()

void loadBuddies ( )

This method requests the buddy-list of the user that is returned by calling getUser().

If no user has been explicitly set, the buddy-list of the session user is requested instead.

After a successful request, access the buddy-list by calling getBuddyUsers() from the User class.

void loadUser ( )

This method requests attribute information for the user returned by calling getUser().

If no user has been explicity set, attributes for the current session user will be requested instead.

See also:
loadUserDetail()
void loadUserContext ( )

This method requests game-specific data for the user who is returned by getUser().

Game-specific data is stored in a java.util.Map and can hold values such as strings, numbers, arrays, dictionaries, or base64-encoded data.

After a successful request, the game-specific data should be accessed via the context property of the user object, (use the object to call User.getContext() to do this).

void loadUserDetail ( )

This method requests both basic data and extended, game-specific details for the user returned by getUser(). If no user has been explicitly set, the details for the session user will be requested instead.

After a successful request, the details can be accessed via the user's detail property, (call the User.getDetail()) from the User class.

This method requests additional statistical information about the user. The loadUser() method should be used if the additional data is not required.

void submitUser ( )

This method is used to push updates of the current session user's attributes to the server.

Only the session user can be updated in this way.

void submitUserContext ( )

This method updates game-specific data for the user. Game-specific user data is stored as a java.util.Map whose values can be strings, numbers, arrays, other maps, or base64-encoded data. The context can also be used to exchange game-specific data between users.

Important:

As the user context can be updated not only for the session user, but for any user, the data is versioned on the server. When trying to update with old data, the server will deny this and return a conflict error (409).

The local data will be updated to reflect the new server data. The user-info property of the error will contain keys with the old (oldUserContext) and new (newUserContext) data.

The application developer is responsible for resolving the conflict and trying the update again.
